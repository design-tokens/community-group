# Design Tokens Community Group \- Role Definitions

**Last updated 12/16/2025**

## Active Contributor

**Definition**: Anyone who has made a meaningful contribution in the last 90 days

**Qualifies through**:

- Being a member of the group (via the W3C site)
- Any of:
  - Submitting a PR (merged or actively reviewed)
  - Substantive comments on issues/PRs\*
- Having attended at least one meeting

\*Substantive comments are determined by the chair's judgment, generally meaning comments that advance technical discussion, identify specific issues, or propose concrete solutions (not just "+1" or general support).

**Rights**:

- Vote on Tier 3 decisions
- Priority consideration for feedback
- Listed as active contributor

**Tracked**: Automatically via GitHub activity \+ meeting attendance

---

## Editor

**Definition**: Core maintainers responsible for spec quality and progress

**Responsibilities**:

- Review Pull Requests within 1 week
- Approve changes based on tier requirements
- Attend monthly editor sync (minimum)
- Own or support at least one spec area

**Authority**:

- Merge Tier 1 changes immediately
- Approve Tier 2-4 changes per framework
- Request changes or block Pull Requests with rationale

**Accountability**:

- Expected response time: 1 week
- Inactive for 90 days → Check-in required
- Inactive for 180 days → Move to emeritus

## Module Lead

**Definition**: Editor who drives progress in a specific spec area

**Responsibilities**:

- Run focused working sessions as needed
- Report progress monthly

**Authority**:

## Chair

**Definition**: Administrative leader focused on operations, not technical decisions

**Responsibilities**:

- Schedule and run general meetings, and ensure all pre-meeting announcements are sent per the Communication and Transparency section.
- Maintain W3C relationship
- Coordinate with vendors and standards bodies
- Ensure process is followed
- Send the monthly communication roll-up to the community via the email list.

**Authority**:

- Set meeting agendas
- Extend timelines for extenuating circumstances
- Break process deadlocks (not technical ones)

**Note**: This is an administrative role. Technical decisions follow the normal framework.

**Term**: 1 year, renewable

## Inactive Status Handling

### Grace Period

- No activity for 90 days → Friendly check-in
- Offer ways to stay involved at lower commitment
- Discuss transitioning responsibilities

### Emeritus Status

- No activity for 180 days → Automatic transition
- Retains recognition and appreciation
- Can return to active status anytime
- Removed from approval counts

---

## **Getting Involved**

### **Becoming an Editor**

- Be an active contributor for 6+ months
- Demonstrate expertise in a spec area
- Nominated by current editor
- **Approved by editor group**: Requires majority vote of current editors via async voting (1-week period) or consensus at editor meeting

### **Becoming a Module Lead**

- Be an active editor
- Show leadership in module area
- Volunteer when position opens
- **Approved by editor group**: Requires majority vote of current editors via async voting (1-week period) or consensus at editor meeting

### **Becoming Chair**

- Nominated by editors or self-nominate
- Community feedback period
- Selected by editor consensus (unanimous agreement preferred, but majority sufficient if consensus cannot be reached)
- 1-year renewable term
